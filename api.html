PPT/PPTX/PDF 转高清图片服务 API 接口规范说明书

运行中 (3000端口) Node.js 22+ 内置 SQLite

对外/外部集成接口 (External API) 外部可用

提供给客户端、主系统或其他微服务调用的接口。用于执行文档上传、自定义 ID 隔离转换、查询具体导出结果及级联删除数据等核心业务流程。

内部服务/环境检测接口 (Internal System API) 系统内部

用于 NodeJS 服务内部状态读取、自带的前端 Web 交互面板加载、或是运维探针对底层运行环境依赖（如 LibreOffice、Poppler 的路径和可用性）的健康巡检接口。

POST /api/upload
对外/外部集成接口

上传 PPT/PPTX/PDF 文件并自动触发高清渲染。对于 PDF 格式，服务将跳过 LibreOffice，实现直通极速转换；支持传递自定义 ID 以确保在物理文件夹级别实现数据隔离。

Request Headers

Content-Type: multipart/form-data

Body Fields (Form Data)

  • file (File, 必填): 上传的 PPT / PPTX / PDF 文档
  • id (String, 可选): 唯一的转换任务 ID。若省略则服务端自动分配 UUID。

Response Sample (200 OK)

{
  "success": true,
  "message": "文件上传并成功转换为图片！",
  "id": "my_custom_task_001",
  "file": "proposal.pptx",
  "pdfPath": "/app/output/my_custom_task_001/proposal.pdf",
  "images": ["slide-1.png", "slide-2.png", "slide-3.png"],
  "baseName": "proposal",
  "outputDir": "/app/output/my_custom_task_001"
}
GET /api/export/:id
对外/外部集成接口

根据设定的任务 ID，导出该任务生成的全部高清 PNG 图片路径与 PDF 副本。常用于主系统（如 NodeJS 服务）在导出操作时直接读取本服务的图档资源链接。

Path Parameters

id: 上传或自动生成的任务唯一 ID 标识。

Response Description

返回 JSON 报文，直接输出每个转换页的真实 Web 静态资源访问 URL 列表 (`imageUrls`) 及 PDF URL。

Response Sample (200 OK)

{
  "success": true,
  "id": "my_custom_task_001",
  "filename": "proposal.pptx",
  "converted": true,
  "outputDir": "/app/output/my_custom_task_001",
  "pdfPath": "/app/output/my_custom_task_001/proposal.pdf",
  "pdfUrl": "/output/my_custom_task_001/proposal.pdf",
  "images": ["slide-1.png", "slide-2.png", "slide-3.png"],
  "imageUrls": [
    "/output/my_custom_task_001/slide-1.png",
    "/output/my_custom_task_001/slide-2.png",
    "/output/my_custom_task_001/slide-3.png"
  ]
}
DELETE /api/delete/:id
对外/外部集成接口

物理删除指定 ID 下在服务器上存储的所有实体数据文件（包含物理上传的源文件、转换生成的全部高清 PNG 图片与 PDF 副本），并在 SQLite 数据库中注销该记录。**支持以 ID 进行精确级联删除，或兼容旧版以文件名删除。**

Path Parameters

id: 任务 ID（或旧版的文件名）。

Safety Features

删除动作将在文件系统物理执行级联清理，清理后不可恢复，有利于释放磁盘空间。

Response Sample (200 OK)

{
  "success": true,
  "message": "删除成功！",
  "deletedSource": true,
  "deletedOutput": true
}
POST /api/convert/:id
对外/外部集成接口

手动重新触发指定 ID（或文件名）的任务进行二次转换和渲染。这也会加入到最大并发数为 2 的全局控制队列中排队，确保核心负载安全。

Path Parameters

id: 任务 ID（或旧版的文件名）。

Scenario

用于物理文件丢失需要重造、或者在对系统配置中的字体加载调整后快速重建图库状态。

Response Sample (200 OK)

{
  "success": true,
  "message": "重新转换成功！",
  "id": "my_custom_task_001",
  "file": "proposal.pptx",
  "pdfPath": "/app/output/my_custom_task_001/proposal.pdf",
  "images": ["slide-1.png", "slide-2.png", "slide-3.png"],
  "baseName": "proposal",
  "outputDir": "/app/output/my_custom_task_001"
}
GET /api/status
内部服务/环境检测接口

检测服务器上运行本转换服务的底层二进制依赖环境。主要由系统巡检服务、K8s 探针或自带的前端 UI 初始化时调用，以检验宿主机环境是否真正就绪。

Check Targets

  • - LibreOffice 无头运行绝对路径、可用性及具体版本
  • - Poppler pdftoppm 工具绝对路径、可用性及版本

Internal Purpose

如果环境存在任何不可用问题（比如 Docker 内命令异常或库缺失），会输出相应的报错明文，指示如何安装依赖。

Response Sample (200 OK)

{

    "available": true,
    "version": "LibreOffice 24.2.2.2 420(Build:2)",
    "error": null
  },
  "poppler": {
    "path": "/usr/bin/pdftoppm",
    "available": true,
    "version": "pdftoppm version 22.02.0",
    "error": null

GET /api/files
内部服务/环境检测接口

拉取 SQLite 数据库中记录的所有文件转换任务的历史流水。主要由服务自带的 Web 控制面板在首次加载时使用，以表格形式将转换状态同步渲染出来。

Database Cache

由于集成了原生 SQLite，读取文件列表不再需要频繁轮询全磁盘，能够大幅削减高并发读取时的 IO 负载开销。

Disk Cross Validation

系统每次都会交叉对比数据库与磁盘的真实物理文件，若物理目录被意外手动删除，会自动动态标记为“未渲染”。
